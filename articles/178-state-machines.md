# ステートマシン：キャラやゲームを『状態』で管理する（待機・歩き・ジャンプ）

ゲームのキャラクターを動かすコードを、素朴なif文だけで書き始めると、あっという間に地獄が訪れる。「ジャンプ中なら攻撃できないが、二段ジャンプ中で壁に触れていて、しゃがみボタンも押されていて……」——条件が掛け算で増え、`if` のネストが崖のように積み上がる。バグの温床であり、誰も読めないコードになる。

この混沌を整理する、驚くほど古典的で強力な道具が **ステートマシン（状態機械）** だ。

## 「いま何をしているか」は必ず1つ

発想はシンプル。キャラクターは常に、はっきり区別できる**ひとつの状態**にいる、と決める。たとえばマリオなら「待機（Idle）」「歩き（Walk）」「ジャンプ（Jump）」「しゃがみ（Crouch）」。ボールが穴に落ちるように、同時に2つの状態にはいられない。

そして状態と状態の間に**遷移（transition）**の矢印を引く。

- 待機中に方向キー → 歩きへ
- 歩き中にAボタン → ジャンプへ
- ジャンプ中に着地 → 待機へ

各状態は「入ったときの処理（enter）」「毎フレームの処理（update）」「出るときの処理（exit）」だけを担当する。ジャンプ状態は重力とジャンプ入力だけを考えればよく、「歩きのことは知らない」。責任が分割されるから、コードが劇的に読みやすくなる。実装上は各状態をクラスにして、現在の状態オブジェクトの `update()` を呼ぶだけ、という形が定番だ。

## キャラだけじゃない、ゲーム全体もそう

この考え方はキャラの動きに限らない。ゲーム全体の流れ——「タイトル画面」「プレイ中」「ポーズ」「ゲームオーバー」——もまさにステートマシンだ。格闘ゲームのコンボ、敵の行動パターン、UIのメニュー階層、信号機、自動販売機まで、世の中の「モードで動くもの」はほぼこれで表現できる。

意外なのは、この概念が1950年代の計算理論（有限オートマトン）に由来する数学的な道具だということ。抽象的な理論が、そのまま現代のアクションゲームのキャラ制御に生きている。

プロの現場では、状態が数十個に膨れると管理が難しくなるため、状態を入れ子にする「階層型ステートマシン（HSM）」や、次回扱う「ビヘイビアツリー」へと発展させる。だが出発点はいつも同じ。**「いま、こいつは何をしている状態か？」を1つに決める**。この一言が、複雑さを飼いならす最初の魔法だ。

## 💡 アプリアイデアの種

- **状態図エディタ**：状態と遷移を丸と矢印でドラッグ配置すると、そのままJavaScript/C#のステートマシン雛形コードを吐き出すノーコードツール。
- **習慣・気分トラッカー**：ユーザーの1日を「集中」「休憩」「移動」などの状態としてログし、遷移パターンから生活リズムを可視化する。
- **業務フロー可視化SaaS**：申請→承認→差し戻しといったワークフローをステートマシンとして定義し、不正な遷移（承認前に完了など）を自動で弾く。
