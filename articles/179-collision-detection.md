# 当たり判定：コンピュータはどうやって『ぶつかった』を知るのか

弾が敵に当たる。マリオがブロックの上に立つ。剣が敵をかすめる。ゲームで「ぶつかった」は当たり前の出来事に見えるが、コンピュータは絵を「見て」判断しているわけではない。裏側では、ひたすら**座標の引き算と大小比較**が走っている。これが当たり判定（コリジョン検出）だ。

## いちばん簡単な四角どうし（AABB）

まず主役たちを、見た目のキャラ絵ではなく**単純な四角形**で近似する。軸に沿った長方形なので「AABB（Axis-Aligned Bounding Box）」と呼ぶ。2つの四角が重なっているかは、たった4つの不等式で判定できる。

```
重なっている = A.left < B.right && A.right > B.left
            && A.top  < B.bottom && A.bottom > B.top
```

X方向でもY方向でも隙間がない、を確認しているだけ。逆に言えば「片方の軸でも離れていれば当たっていない」。丸いキャラなら、もっと簡単に「中心どうしの距離が半径の合計より近いか」で判定できる（円判定）。距離の計算では、平方根が重いので**距離の2乗どうしを比べる**のがプロの定番の高速化テクだ。

## 10000個をどうさばくか

問題は数だ。敵100体、弾500発、地形ブロック多数……全ペアを総当たりでチェックすると計算量は個数の2乗で爆発する。1000個あれば約50万回、これを毎フレーム(秒60回)は現実的でない。

そこでプロは2段構えにする。まず**ブロードフェーズ**で「そもそも近くにいる候補」だけをざっくり絞る。画面をマス目に区切る「空間グリッド」や、領域を再帰的に4分割する「四分木（Quadtree）」を使い、遠く離れたペアは最初から比較しない。そのうえで残った少数だけを**ナローフェーズ**で厳密に判定する。この工夫で、シューティングの弾幕やパズルの大量オブジェクトが破綻せず動く。

## 「すり抜け」という悪夢

高速な弾には別の罠がある。1フレームで大きく動くと、あるフレームでは敵の手前、次のフレームでは敵の向こう側にワープし、**間の壁をすり抜ける**（トンネリング現象）。離散的に世界を更新するゲームの宿命だ。対策は、点ではなく「前の位置から今の位置まで引いた線分」で交差を調べる連続的衝突判定（CCD）や、レイキャスト（光線を飛ばして最初に当たる物を探す手法）。FPSの射撃判定はまさにこれで、引き金を引いた瞬間に見えない線が敵に届いたかを計算している。

「ぶつかった」というアナログな体験が、実は不等式と幾何学の積み重ねだと知ると、ゲームが少しだけ違って見えてくる。

## 💡 アプリアイデアの種

- **当たり判定ビジュアライザ**：AABB・円・四分木の絞り込みをアニメで見せ、総当たりとの計算回数を並べて表示する学習アプリ。
- **ARの接触検知ミニゲーム**：スマホカメラで現実の物体を四角/円で近似し、仮想オブジェクトとの衝突で反応する子ども向け知育アプリ。
- **地図アプリの近接通知**：空間グリッドで「近くの候補だけ」を高速抽出し、大量のスポットから徒歩圏内の店だけをリアルタイム表示する省電力ロジック。
