# プロトタイピングとゲームジャム：早く作って早く試す（面白さは触るまで分からない）

「頭の中では最高に面白いゲーム」が、実際に作って触ってみたら退屈だった。ゲーム開発で最も多い悲劇はこれだ。面白さは、企画書やスプレッドシートの上には存在しない。指でコントローラーを握り、キャラクターがジャンプした瞬間の「気持ちよさ」の中にしかない。だから熟練の作り手ほど、議論より先に「動くもの」を作る。

任天堂の宮本茂は、マリオの操作感を「触った感触（さわり心地）」と呼び、企画のごく初期から動くプロトタイプで確かめる文化を築いたことで知られる。ジャンプの高さ、落下の速さ、着地の重みを、ミリ秒単位で何百回も調整する。これは数値を眺めていても決して分からない。実際に飛ばして、着地して、「うーん、なんか違う」と感じるまで確かめるしかない領域だ。

この「早く作って早く試す」を極限まで凝縮したイベントが**ゲームジャム**である。48時間や72時間という制限の中で、テーマ発表を受けて即興でゲームを1本完成させる。世界最大級の「Global Game Jam」には毎年数万人が参加する。ここでの合言葉は「完成度より完成」。汚くていい、バグがあっていい、まず遊べる状態まで持っていくことに全力を注ぐ。

驚くのは、この即興の産物が化けることがある点だ。人気ゲーム『Superhot』（自分が動くときだけ時間が進むFPS）は、もとは7日間のゲームジャム「7DFPS」で生まれた試作だった。『Goat Simulator』も社内ジャムのおふざけプロトタイプが、あまりに笑えたので製品化された。逆に言えば、何ヶ月も企画を練り込んでいたら、こんな「バカバカしいけど面白い」着想は磨かれる前に理屈で却下されていただろう。

プロトタイピングの本質は「捨てる前提で作る」ことにある。この段階のコードやグラフィックは、後で全部捨ててもいい。目的は完成品を作ることではなく、「この遊びの核（コアメカニクス）は本当に面白いのか？」という問いに、最短で答えを出すことだ。紙とペンで作る「ペーパープロトタイプ」で十分なことも多い。ボードゲームのようにコマを手で動かして、ルールの骨格だけを検証する。

早く作って早く試す。失敗を早く、安く済ませる。頭の中の「面白いはず」を、指先の「面白い！」に変えられるかどうか。それを確かめる勇気こそが、企画を製品に近づける第一歩なのだ。

## 💡 アプリアイデアの種

- ゲームジャム向けの「制限時間タイマー＋テーマガチャ」アプリ。開始時にランダムなお題（例：「重力」「たった1つのボタン」）を引き、残り時間を大きく表示。途中の進捗写真を時系列で記録し、後から振り返れる。
- ペーパープロトタイプ支援ツール。スマホのカメラで手描きのコマや盤面を撮り、ルールをテキストで記述しておくと、コマの移動をタップで記録して「実際に遊んだログ」を残せる。デジタル化前の検証を軽くする。
- 「捨てる前提のプロトタイプ置き場」アプリ。作りかけの試作を短い動画とひとことメモで大量にストックし、「これは化けそう」だけを後で拾い上げる。判断は理屈でなく「触った感触の記憶」で行えるよう、当時の手応えを5段階で残す。
