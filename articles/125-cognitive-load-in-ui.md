# 認知負荷：UIがユーザーの脳を疲れさせる仕組み

私たちの脳には「ワーキングメモリ」という短期の作業台がある。ここに一度に置けるものは、驚くほど少ない。かつて心理学者ミラーは「7±2個」と言ったが、その後の研究では実質的に「4個前後」しか同時に扱えないとされる。電話番号を復唱できても、いきなり16桁のクレジットカード番号を一度で覚えるのが難しいのは、この作業台がすぐ満杯になるからだ。

UIデザインの世界では、この作業台にかかる負担を「認知負荷」と呼ぶ。そして認知負荷には3つの種類がある。ひとつ目は**課題内在性負荷**——タスクそのものの難しさ。確定申告は本質的に難しい。ふたつ目は**課題外在性負荷**——デザインの下手さが生む余計な負担。ボタンがどこにあるか探す、専門用語の意味を推測する、この種の「本来なくてよかった」負担だ。三つ目は**学習関連負荷**——理解や上達につながる良い負担。

デザイナーが減らすべきは、まぎれもなく二つ目の外在性負荷である。ここに面白い実験がある。ある研究で、同じ内容を「読みにくいフォント」で提示したところ、被験者はむしろ内容をよく記憶していた——という結果が話題になった。だが実務では逆で、決済フォームの入力エラーが1つ増えるごとに離脱率が跳ね上がる。人は「考えさせられる」瞬間に、そっとタブを閉じる。

外在性負荷を減らす具体策はシンプルだ。選択肢を減らす。専門用語を日常語に言い換える。関連する情報を近くに置く。すでに入力した情報を二度聞かない。郵便番号から住所を自動補完するだけで、ユーザーの作業台には空きが生まれる。名著『Don't Make Me Think(考えさせるな)』のタイトルは、この一点に尽きる。

覚えておきたいのは、認知負荷は「見えない疲労」だということ。ユーザーは「このアプリ、なんか疲れる」としか言語化できない。だが裏側では、脳の小さな作業台が悲鳴を上げている。良いUIとは、機能が多いことではなく、ユーザーの脳に余白を残すことなのだ。

## 💡 アプリアイデアの種

- 案1: 自作のUI画面をアップすると、選択肢の数・専門用語・入力項目から「認知負荷スコア」を算出し、削れる箇所をハイライトするデザイン診断ツール。
- 案2: フォーム入力中のユーザーの逡巡(入力の中断・削除・再入力)を計測し、どのフィールドが脳を疲れさせているかをヒートマップ表示する分析SDK。
- 案3: 業務マニュアルや契約書を貼ると、ワーキングメモリの限界(4項目)を超える段落を検出し、箇条書きや分割を提案する「読みやすさ最適化」エディタ。
