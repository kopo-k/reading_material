# オンボーディング：最初の5分でユーザーは去るか残るか決める

アプリを初めて開いた瞬間、ユーザーの頭の中には無言の問いが3つ浮かんでいる。「これは何?」「自分の役に立つ?」「どう使えばいい?」。この3つに数分で答えられなければ、その人は静かに去る。しかも一度去ったユーザーは、めったに戻ってこない。モバイルアプリの世界では、インストールした人の多くが最初の数日以内に使うのをやめ、1週間後に残るのはごく一部、という厳しい数字がよく引かれる。最初の体験——オンボーディング——は、それほど生死を分ける。

やりがちな失敗は、いきなり10画面のチュートリアルを見せることだ。「次へ」「次へ」と押させる説明の連続は、たいてい飛ばされる。人は説明書を読みたいのではなく、早く「できた!」を体験したいのだ。ここで鍵になるのが「アハ体験(Aha moment)」——ユーザーが初めて価値を実感する瞬間だ。あるSNSは「早い段階で一定数の友達とつながったユーザーは定着する」という法則を見つけ、オンボーディング全体をその一点に集中させたことで知られる。数を集めることではなく、価値を最短で味わわせることが目的なのだ。

優れたオンボーディングの共通点は3つある。ひとつ、**登録の摩擦を減らす**。最初から全項目を入力させず、まず使わせて後から埋めさせる。ふたつ、**説明より体験**。「見せる」より「やらせる」。実際のデータを1件だけ一緒に作ってみる。みっつ、**空っぽで放置しない**。真っさらな画面に「さあどうぞ」と置き去りにせず、サンプルや次の一手を用意する。

心理学的な後押しもある。人は途中まで進んだ物事を完成させたくなる(ツァイガルニク効果)。プロフィール完成度を「70%」と示すだけで、残りを埋めたくなる。チェックリストが1つずつ埋まっていく設計は、この心理を借りている。

オンボーディングは「機能の説明会」ではない。初対面の相手に、短い時間で「あなたの生活はこう良くなる」と体験させる場だ。最初の5分は、宣伝文句100個より雄弁に、そのプロダクトの価値を語る。

## 💡 アプリアイデアの種

- 案1: 新規ユーザーの初回セッションを解析し、「アハ体験」に到達した人と離脱した人の行動の分岐点を可視化するオンボーディング分析ダッシュボード。
- 案2: サンプルデータ自動生成つきの初期設定ウィザードを、どんなSaaSにも数行で組み込めるオンボーディング用SDK。
- 案3: 完成度バーとチェックリストでプロフィール入力を段階誘導し、ツァイガルニク効果で自然に設定を進めさせる汎用オンボーディングUIキット。
