# ゲームループ：毎秒60回、世界を更新して描き直す心臓部（デルタタイム）

ふだん遊んでいるゲームは、実はとんでもない回数の「やり直し」で成り立っている。マリオが左から右へ滑らかに走っているように見えても、画面の裏では**1秒間に60回、世界がまるごと計算し直されている**。この心臓の鼓動が「ゲームループ」だ。

ゲームループの正体は、ものすごく速く回る `while(true)` の無限ループにすぎない。そのなかで、だいたい3つのことを繰り返している。

1. **入力を読む**（Input）：ボタンは押されたか、スティックはどっちに倒れているか
2. **世界を更新する**（Update）：敵を動かし、重力をかけ、当たり判定をとる
3. **描き直す**（Render）：更新後の世界を1枚の絵にして画面に出す

この「Input → Update → Render」を1周する時間が短いほど、動きは滑らかになる。60回転すれば「60fps（フレーム毎秒）」、1周あたり約16.7ミリ秒。人間の目には連続した動きに見える、パラパラ漫画の超高速版だ。

## デルタタイムという発明

ここで初学者がハマる落とし穴がある。素朴に「毎フレーム、敵を1ピクセル動かす」と書くとどうなるか。60fpsのPCでは秒速60ピクセル、120fpsの高性能PCでは秒速120ピクセル。**同じゲームなのに、速いマシンほどキャラが倍速で動いてしまう**のだ。昔のPCゲームで「新しいPCだと速すぎてクリア不能」という珍事が起きたのは、まさにこれが原因だった。

これを解決するのが **デルタタイム（前フレームからの経過秒数、`dt`）** だ。移動を「1フレームで1ピクセル」ではなく「1秒で60ピクセル」と時間ベースで書く。

```
position += velocity * dt;  // dt は例えば 0.0167 秒
```

こうすれば、フレームレートが変動しても、1秒後には必ず同じ距離だけ進む。マシンの速さに関係なく、ゲーム内の時間が現実の時間と一致する。プロのエンジンでは、さらに物理演算だけを固定間隔（例：秒120回）で回す「固定タイムステップ」を使い、描画とは別リズムで動かして安定させている。

面白いのは、私たちが「なめらかさ」と呼ぶ感覚が、実は**離散的な計算のごまかし**だということ。世界は連続して動いているのではなく、16ミリ秒ごとにワープしている。それでも脳が勝手に補間して、連続した運動として認識する。ゲームとは、脳の錯覚を60分の1秒単位で設計する芸術なのだ。

## 💡 アプリアイデアの種

- **可視化学習アプリ**：fpsを10・30・60・120と切り替え、デルタタイムあり/なしで同じボールを転がして「速すぎバグ」を体感できる教材。プログラミング初学者向け。
- **タイムベース設計のタスク管理**：習慣アプリを「フレーム（1日）」ではなく「経過時間」で進捗計算し、更新間隔がバラついても目標ペースを崩さないロジックに。
- **リズムゲームのフレーム精度チェッカー**：端末ごとの描画間隔を計測し、判定タイミングのズレを補正するキャリブレーション機能。
