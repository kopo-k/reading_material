# reading_material — 知識インプット用 読み物ライブラリ

## プロジェクトの目的

オーナーの知識がプログラミングに偏っているため、**分野横断の短編エッセイ**を読んで視野を広げ、
アプリ開発の着想（アイデア）につなげることが目的。読み物を通じて「知らない分野の視点」を
継続的にインプットするためのリポジトリ。

## ディレクトリ構造

```
reading_material/
├── CLAUDE.md          # このファイル（運用ルール）
├── README.md          # 読み物インデックス（分野別リンク一覧）
└── articles/
    ├── 001-<slug>.md  # 連番3桁 + 内容を表す英小文字スラッグ
    ├── 002-<slug>.md
    └── ...
```

## 記事フォーマット

各記事は `articles/` 配下に、以下のフォーマットの Markdown で作成する。

```markdown
---
title: 記事タイトル（日本語）
category: 分野名
reading_time: 約4分
date: YYYY-MM-DD
---

# 記事タイトル

本文（800〜1500字の短編エッセイ。3〜5分で読める分量。
専門用語は噛み砕き、平易で親しみやすい語り口にする。
1つの面白い切り口・気づきを軸にまとめる）…

## 💡 アプリアイデアの種

- 案1: この知識から着想できるアプリ・機能のヒント
- 案2: …
- 案3: …
```

### フォーマット規約
- **frontmatter 必須**: `title` / `category` / `reading_time` / `date` の4項目。
- **本文**: 800〜1500字。読者を退屈させない語り口。数字・具体例・意外性を盛り込む。
- **「💡 アプリアイデアの種」欄は必須**。その記事の知識から着想できるアプリ案を2〜4個。
- すべて**日本語**で書く。

## 採番ルール

- ファイル名は `NNN-<slug>.md`（3桁ゼロ埋め連番 + 英小文字スラッグ）。
- 新規追加時は `articles/` の既存最大番号の次から採番する。
- 番号の重複を避けること。

## 追加生成のやり方

- 読み物を**追加で量産する際は、必ずサブエージェントを並列で使う**（重い処理はサブエージェントに任せる）。
- 分野ごと・番号レンジごとにサブエージェントを割り当て、番号衝突を避ける。
- 生成後は必ず `README.md` のインデックスを更新する。

## 分野の例（幅広くおまかせ）

歴史・文明 / 心理・脳科学 / 生物・自然 / 経済・行動経済学 / 物理・数学 /
哲学・思想 / 芸術・デザイン / 言語・文化人類学 / テクノロジーと社会 / 日常の科学・雑学

## Git 運用

- このリポジトリは Git 管理（origin: GitHub）。**変更があればコミットして push する**。
- コミットメッセージに `Co-Authored-By` を含めない（AI を共著者として記載しない）。
- PR を勝手にマージしない（マージはユーザーが手動で行う）。

## 応答言語

- オーナーとのやり取り、記事、コメントはすべて**日本語**で行う。
